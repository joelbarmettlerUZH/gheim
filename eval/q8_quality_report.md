# q8 ONNX quality report — `joelbarmettler/gheim-ch-560m`

_Generated by `eval/q8_quality_eval.py`. Probe: 212 forensic cases, uncalibrated (`o_bias=0`), Python `onnxruntime` CPUExecutionProvider, no chunking. Eval script is auditable; raw per-case results in `eval/q8_quality_per_case.json`._

## Headline

**The "18.4pp q8 gap" reported in `calibration_sweep_q8.json` is not a quantisation problem — it is a `transformers.js` / WASM runtime problem.** Running the same 212-case probe through Python `onnxruntime` (CPU provider) gives essentially identical results across all three dtypes: fp32 91.5%, fp16 91.5%, q8 91.0%. The two q8 smoke-test failures from `/tmp/gheim_smoke.mjs` (Bach false positives; Beat Aregger / Sandra Fritsche miss) **do not reproduce** in Python on the same `model_quantized.onnx` file — they only appear via the `@huggingface/transformers` / ONNX-Runtime-Web execution path. Diagnostically: the int8 dynamic quantisation Optimum produced is sound; the kernel that consumes it in `transformers.js` is the failure mode. fp16 closes 100% of the (tiny) Python-side gap and never produces a worse argmax than fp32 on any of the 212 cases.

## Numerical noise (sanity check)

Per-token logit divergence on 104 BPE positions across 10 probe cases:

| dtype | mean abs logit diff vs fp32 | per-token argmax-match vs fp32 |
|---|---:|---:|
| fp16 | 0.0011 | 104/104 (100.0%) |
| q8   | 0.1627 | 104/104 (100.0%) |

q8 carries ~150× more numerical noise than fp16, but on this sample the noise is small enough not to flip any per-token argmax — the post-aggregation span result is identical to fp32. fp16 is essentially bit-equivalent. This confirms the int8 file is not the failure mode.

## Diagnosis

The Optimum-produced `model_quantized.onnx` is fundamentally sound. Inside Python `onnxruntime` (the same C++ kernels Optimum was designed against), q8 gives near-fp32 numbers on the probe and matches fp32 exactly on the two smoke-test cases that were reported as broken. The 18pp difference and the smoke-test fails come from running that same file through `transformers.js` (which uses `onnxruntime-web` WASM kernels, a different aggregation pipeline, and a different tokenizer implementation). Likely candidates inside transformers.js:

1. **WASM int8 kernel differences vs the C++ AVX-512-VNNI kernels** that Optimum's `avx512_vnni(per_channel=True)` config targets. The shipped `model_quantized.onnx` is per-channel int8 — `onnxruntime-web` may fall back to per-tensor handling or a slower reference kernel that rounds differently.
2. **`aggregation_strategy="simple"` reimplemented in JS** with subtly different boundary semantics (BIOES collapse rules) than the Python pipeline. Our Python harness re-implements aggregation from scratch matching the CalibratedDetector logic, which matches fp32 PyTorch.
3. **Tokenizer normalization differences** (`@huggingface/tokenizers` JS uses its own JS implementation, not the Rust binding). Small offset shifts could push entity boundaries off the gold span by one BPE piece, which the strict-span verdict counts as PARTIAL/FAIL.

Confirming which of these is the root cause is a separate investigation, but it is **a JS investigation, not a quantisation investigation**.

## Recommendation

**Ship fp16 alongside q8, and change the SDK default to fp16 when the runtime supports it (option a + c, blended).**

The reasoning:

1. **The current q8 file is fine for Python users**. The 91.0% Python-side number is within stochastic noise of fp32. Anyone using `gheim-py` via `onnxruntime` is not affected by the reported gap. Keep `model_quantized.onnx` on the Hub; don't break that path.

2. **Browser/Node users hit the JS-WASM-q8 pathology**. They lose 18pp of probe correctness on the same checkpoint, including specific high-value Swiss patterns (commercial-register person names, common-word surnames). That's not acceptable as the default for the only browser-deployable build. Fixing transformers.js's int8 path is upstream work we don't control.

3. **fp16 demonstrably bypasses the issue**: it's byte-equivalent to fp32 on the probe (194/212 perfect, identical case-by-case to fp32), it halves the download (1.1GB vs 2.2GB) compared to fp32, and `onnxruntime-web` supports fp16 in both WebGPU and the standard WASM SIMD backend. Users with capable hardware get the right answers; users on phones who can't afford 1.1GB still have q8 as a fallback.

4. **Concrete actions** (in order):
   - (a) Re-export fp16 via the patched converter at `eval/regenerate_fp16_onnx.py`, then upload `checkpoints/gheim-ch_onnx_fp16_v2/model.onnx` + `model.onnx_data` to `joelbarmettler/gheim-ch-560m` under `onnx/model_fp16.onnx`. The previously-uploaded fp16 (and the local `checkpoints/gheim-ch_onnx_fp16/`) had mixed-precision Div/Mul/MatMul/LayerNormalization type errors and is unloadable in any onnxruntime — that's why this report wasn't written months ago. Fold `regenerate_fp16_onnx.py`'s logic back into `training/src/gheim_training/eval/quantize_onnx.py::quantize_fp16()` so future re-exports just work.
   - (c) Change `gheim-js` `LocalDetector` default from `dtype: "q8"` to `dtype: "fp16"` when `device !== "cpu"` (i.e. WebGPU available), keeping `q8` as the explicit fallback. Document the gap in the JS package README.
   - **Do not** remove `model_quantized.onnx` — keep it for low-RAM mobile fallback, since the failure mode is severe but the file still works for non-Swiss content and for Python users.
   - File an upstream issue against `transformers.js` referencing the per-channel int8 + Optimum `avx512_vnni` config and the specific failing cases. Don't block our ship on it.

The model card should additionally note the JS-q8 caveat: "When running this model via `transformers.js` on int8, expect a ~18pp drop on Swiss forensic probe versus the underlying ONNX file; use `dtype: 'fp16'` if your runtime supports it."

This is the minimum-risk path: it fixes the actually-broken deployment (browser q8) by routing around it, ships the high-precision fp16 build that should always have existed (the previous one was unloadable), and doesn't disrupt Python users who are unaffected.
